# Найденные баги API

## BUG-001. Пустое тело запроса приводит к Internal Server Error

**Эндпоинт:** `POST /booking`

### Шаги воспроизведения

1. Отправить запрос `POST /booking`.
2. Не передавать тело запроса.
3. Выполнить запрос.

### Ожидаемый результат

Сервер возвращает:

```http
400 Bad Request
```

с сообщением о том, что обязательные поля отсутствуют.

### Фактический результат

Сервер возвращает:

```http
500 Internal Server Error
```

### Серьёзность

Medium

### Комментарий

Некорректный клиентский запрос вызывает внутреннюю ошибку сервера вместо ошибки валидации.

---

## BUG-003. API принимает строковое значение в поле totalprice

**Эндпоинт:** `POST /booking`

### Шаги воспроизведения

1. Передать в поле `totalprice` значение:

```json
{
  "totalprice": "abc"
}
```

2. Выполнить запрос.

### Ожидаемый результат

Сервер возвращает:

```http
400 Bad Request
```

с сообщением о неверном типе данных.

### Фактический результат

Сервер возвращает:

```http
200 OK
```

Бронирование успешно создаётся.

Значение поля сохраняется как:

```json
"totalprice": null
```

### Серьёзность

High

### Комментарий

Система не выполняет валидацию типа данных и сохраняет некорректное значение.

---

## BUG-004. Можно создать бронирование с датой выезда раньше даты заезда

**Эндпоинт:** `POST /booking`

### Шаги воспроизведения

1. Передать даты:

```json
{
  "checkin": "2026-01-01",
  "checkout": "2025-01-02"
}
```

2. Выполнить запрос.

### Ожидаемый результат

Сервер возвращает:

```http
400 Bad Request
```

с сообщением об ошибке валидации дат.

### Фактический результат

Сервер возвращает:

```http
200 OK
```

Бронирование успешно создаётся.

### Серьёзность

High

### Комментарий

Дата выезда не может быть раньше даты заезда. Отсутствует проверка бизнес-логики.

**Эндпоинт:** `POST /booking`

### Шаги воспроизведения

1. Передать даты:

```json
{
  "checkin": "2018-01-40",

```

2. Выполнить запрос.

### Ожидаемый результат

Сервер возвращает:

```http

```

с сообщением об ошибке формата даты.

Сервер возвращает:


## BUG-006. PUT/PATCH/DELETE с несуществующим ID возвращают 405 вместо 404

**Эндпоинты:** `PUT /booking/:id`, `PATCH /booking/:id`, `DELETE /booking/:id`

### Шаги воспроизведения

1. Получить токен через `POST /auth`.
2. Отправить запрос на несуществующий ID, например:

```http
PUT /booking/99999999
Cookie: token=<valid_token>
```

3. Выполнить запрос.

### Ожидаемый результат

Сервер возвращает:

```http
404 Not Found
```

с сообщением о том, что бронирование не найдено.

### Фактический результат

Сервер возвращает:

```http
405 Method Not Allowed
```

То же поведение воспроизводится для `PATCH` и `DELETE`.

### Серьёзность

Medium

### Комментарий

405 указывает на то, что метод не поддерживается ресурсом, но `PUT`, `PATCH` и `DELETE` для `/booking/:id` поддерживаются. Корректный ответ — `404 Not Found`. Текущее поведение вводит в заблуждение клиентов API и затрудняет обработку ошибок.

# Наблюдения

## OBS-001. API позволяет создавать бронирования с датами в прошлом

**Эндпоинт:** `POST /booking`

### Шаги воспроизведения

1. Передать даты:

```json
{
  "checkin": "2000-01-01",
  "checkout": "2000-01-02"
}
```

2. Выполнить запрос.

### Ожидаемый результат

Зависит от бизнес-требований системы.

Если создание исторических бронирований не предусмотрено, сервер должен вернуть ошибку валидации.

### Фактический результат

Сервер возвращает:

```http
200 OK
```

Бронирование успешно создаётся.

### Комментарий

Без описанных бизнес-требований невозможно однозначно определить, является ли данное поведение дефектом. Поэтому оно оформлено как наблюдение.
